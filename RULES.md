RULES.md

Here I present rules that you must follow to produce great outputs to me, things I care or how I think or like to do stuff.

1. I prefer R for data analys, since that is the programming language I know most.
2. I prefer python for machine learning, web scrapping and text processing because it is better suited to such tasks.
3. I prefer ptyhon for web apps or CLI models or apps.
4. I trust what has been tested. I am skeptical of what hasn't been tested.
5. For papers and reports, the desired default output is pdf.
6. Always start with a Q&A. 
7. Follow up with more questions after the first iteration of Q&A, but a smaller number of questions. Be straight to the point.
8. More often than not I have an ultimate or long term goal than what I am asking for in a given task. If helpful to complete the present task, try to confirm what it is.
9. Keep computation in separate R scripts.
10. Make reports, analysis and academic papers fully reproducible.
11. Preserve raw/extracted files in the repo
12. Ao realizar validações de dados, cheque regras lógicas de datas e valores incompatíveis com o que se espera de variáveis, quando aplicável.
13. Always push back on my bad ideas. If you have reason to believe it is not sound, push back.
14. I know abut causal inference, statistics, impact evaluation, polisci and IR. Not so much about apps.
15. Remove ambiguity over power: any label or flow that can be misread gets replaced by plain language.
16. In an app, make the system explain itself: if a step requires hidden system knowledge (upload vs attach), the UI must either explain it or eliminate the extra step.
17. Bias toward actionability: buttons should read as verbs (“Unlock session”), not states (“Session locked”).
18. Reduce cognitive load. Remove redundant options
19. Bold identity, but airy space: strong visual theme (neo‑brutalist) with generous spacing to avoid overwhelming dense workflows.
20. Minimize steps for common paths
21. Naming: replace jargon with functional nouns
22. Visual hierarchy by weight: bold headers for identity + action; smaller supporting text for status/hints.
23. Avoid decorative friction: no shadows on black buttons; keep them clean and authoritative.
24. Feedback explícito de execução: toda ação longa mostra estado (“Running…”, “Completed”)
25. Bloqueio visível e irrecusável: quando algo está locked, o CTA vira estado inativo e o input fica desabilitado.
26. UI deve eliminar passos invisíveis: se o sistema exige “attach/index”, isso tem que ser automático ou explicado no próprio controle.
27. Evitar metadados irrelevantes: esconder campos que não mudam a decisão (ex.: domain/method no review).
28. Títulos e botões como ações: verbos claros (“Run Review”), nunca labels ambíguos ou estados.
